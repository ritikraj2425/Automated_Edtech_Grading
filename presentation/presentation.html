Section 01

The Evolution of the System

We started with basic tools but soon realized handwritten text and meaning evaluation are highly complex. Here's the journey to Phase 3.

Early Prototypes
Tesseract & TF-IDF

Initially we used Tesseract for OCR and TF-IDF for scoring. Tesseract failed miserably on messy handwriting, and TF-IDF only matched exact words, unfairly penalizing students who paraphrased perfectly valid answers.

Final Solution
Transformer Models: TrOCR & Sentence-BERT

We upgraded to Microsoft's TrOCR, a transformer specifically trained for handwriting. For grading, we implemented Sentence-BERT which understands semantic meaning. Now "loans were provided" and "banks gave credit" score highly despite having no overlapping words.

Final Addition
Gemini API Benchmarking

To ensure our AI scoring is fair, we integrated Google's Gemini as an independent evaluator. The system passes the image and teacher answers to Gemini and displays a side-by-side comparison of our marks vs. the LLM's marks.

Section 02

What each tool does

Block detection
OpenCV + Morphological Ops

Uses horizontal projection to find gaps between answers. Newly added line segmentation further splits blocks into individual lines to optimize OCR accuracy.

Text reading
Microsoft TrOCR

Replaces Tesseract. A state-of-the-art Vision-Encoder-Decoder model that reads highly cursive and messy handwriting with incredible accuracy.

Early Prototypes
Tesseract OCR

Used initially. It struggled heavily with non-standard handwriting and was retired from the pipeline.

Semantic Scoring
Sentence-BERT (SBERT)

Calculates meaning-based cosine similarity. It is combined with keyword overlap metrics to ensure both conceptual understanding and factual correctness.

Benchmarking
Google Gemini LLM

Acts as a secondary AI examiner. Receives the raw image and provides reasoning and marks for comparison against our local system.

Web server & Pipeline
Flask + pdf2image

Serves the UI, handles multi-page PDF uploads, coordinates the models globally at startup to eliminate latency, and orchestrates the data flow.

Section 04

Handling Multiple Questions

Our system handles sheets with multiple answers cleanly and efficiently without manual cropping.

How it works now

The OCR system uses Regex pattern splitting. It automatically searches for labels like "Q1", "1)", or "Answer 2" at the start of segmented lines.

Even if a student writes multiple answers inside a single visual block (without much spacing), the system reads the text, identifies the new question label, and dynamically splits the block textually. Each answer is then routed to its respective model answer for an isolated score.

# Extracted Block Text: Q1: Green revolution was set up by the government. Q2: The banks provided necessary loans. # System Auto-Routes to: Score for Q1: 8.5/10 Score for Q2: 9.0/10
Section 05

Current Limitations

Limitation
TrOCR Computational Overhead

TrOCR is a large transformer model. Because we perform line-by-line inference, processing a densely written page can take 5-15 seconds on a CPU.

Limitation
Gemini API Key Required

The benchmark feature requires a valid Google Gemini API key. Without it, the system falls back gracefully but the comparison feature is disabled.

Limitation
Extreme Lighting Conditions

While adaptive thresholding helps, extremely dark photos with harsh shadows can still merge whitespace, making block detection unreliable.
